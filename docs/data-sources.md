# Data sources

BuildingLens uses public, reproducible data only. Everything here can be re-fetched from scratch by the pipeline (`make data`). We combine two real public Luxembourg sources with a synthetic inspection-report layer, and we are explicit about what is real and what is generated.

Details below were verified on 2026-06-17.

| Source | Type | Role in BuildingLens | License | Reproducible |
|---|---|---|---|---|
| EUBUCCO v0.2 (Luxembourg) | Structured, per building | Real building stock: footprints, height, location | ODbL (LU subset: CC0) | Yes, S3 download by script |
| STATEC "Autorisations de bâtir" | Structured, aggregated | Sector context: permit trends by canton and type | CC0 | Yes, LUSTAT SDMX API |
| Synthetic inspection reports (LU) | Unstructured, PDF | Inspection corpus for extraction and RAG | Generated by us | Yes, generated by script |

## 1. EUBUCCO v0.2 (Luxembourg buildings)

Real building footprints for the whole country. This is our per-building anchor: it populates the `buildings` table with real geometry, height, and location.

- **What it is:** European building stock database, 200M+ individual buildings. Luxembourg is one NUTS2 region, code `LU00`. The v0.1 paper reports 143,923 buildings for Luxembourg.
- **Publisher:** ai4up project (MCC Berlin, TU Berlin). Data paper: Milojevic-Dupont, Wagner et al., *Scientific Data* 10, 147 (2023), DOI 10.1038/s41597-023-02040-2.
- **Homepage:** https://eubucco.com
- **Docs:** https://docs.eubucco.com/v0.2/
- **License:** ODbL v1.0 for most of the dataset. The Luxembourg source (`gov-luxembourg`, the national LOD1 3D building base from the cadastre) is CC0. We still credit EUBUCCO as good practice.

### Download (no account, anonymous S3)

Luxembourg Parquet file (geometry as WKB, EPSG:3035):

```
https://s3.eubucco.com/eubucco/v0.2/buildings/parquet/nuts_id=LU00/LU00.parquet
```

Script-friendly options: direct `curl`/`wget` on that URL, anonymous AWS CLI (`--no-sign-request --endpoint-url https://s3.eubucco.com`), GeoPandas `read_parquet` with `storage_options={"anon": True, ...}`, or a remote DuckDB `read_parquet` query. Estimated Luxembourg file size is 20 to 40 MB (derived, not officially listed).

### Fields and Luxembourg coverage

| Field | Available for LU | Note |
|---|---|---|
| `id` | Yes | Stable EUBUCCO id |
| `geometry` (footprint) | Yes (100%) | EPSG:3035, reproject to get lat/lon |
| `height` | Yes (100%) | Ground truth from 2013 photogrammetry |
| `type` / `subtype` | Estimated | ML-estimated in v0.2, check confidence fields |
| `construction_year` | No (0% ground truth) | Source has no year data; any value is an ML estimate |
| building name | No | EUBUCCO has no names |
| address | No | EUBUCCO has no addresses |

### Mapping to the `buildings` table

| Column | From EUBUCCO | Status for LU |
|---|---|---|
| `id` | `id` | Direct |
| `latitude` / `longitude` | centroid of `geometry` reprojected to EPSG:4326 | Direct |
| `source` | `geometry_source` (`gov-luxembourg`) | Direct |
| `height_m` | `height` | Real, 100% |
| `year_built` | `construction_year` | Mostly empty or estimated, treat as unreliable |
| `name`, `address` | none | Not available, see "Building identity" note below |

### Key caveat: no names, no addresses, no reliable year

EUBUCCO gives us real geometry, height, and coordinates, but **no building names, no addresses, and essentially no construction year for Luxembourg**. Heights come from a single 2013 photogrammetric epoch, so buildings built after 2013 may be missing or interpolated. This shapes how we present building identity in the demo (documented in the README and in the synthetic-data note below).

## 2. STATEC "Autorisations de bâtir" (building permits)

Official Luxembourg construction-permit statistics. Used as **sector context**, not as per-building data.

- **Dataset page:** https://data.public.lu/en/datasets/entreprises-construction-et-logement-autorisations-de-batir/
- **Publisher:** STATEC (national statistics institute), served from the LUSTAT database.
- **License:** CC0 (public domain).

### Access: LUSTAT SDMX REST API (no key)

There are no direct CSV files on the portal; the resources point to LUSTAT. Machine-readable data comes from the SDMX REST API:

```
https://lustat.statec.lu/rest/data/LU1,{DATAFLOW_ID}/all?startPeriod={YYYY}
```

Send header `Accept: application/vnd.sdmx.data+csv;urn=true;file=true;labels=both` to get labeled CSV. Useful dataflows:

| Dataflow | Content | Level |
|---|---|---|
| `DF_D4100` | Buildings, dwellings, built volume (overview) | National, 1970-2022 |
| `DF_D4113` / `DF_D4123` | Dwellings / buildings authorized by type | Canton, 2010-2025 |
| `DF_D4131` to `DF_D4133` | Authorized useful floor area (thousands of m2) | National / regional / canton |

Granularity goes down to the 12 cantons (no commune level). Permit counts are annual; floor area is also quarterly.

### Caveats

- **Aggregated, not per-building.** Every row is a count or surface total by type, canton, and year. No permit id, address, or parcel. You cannot identify individual buildings.
- **Permits, not inspections.** Authorization to build says nothing about later inspection or defects.
- **Classification changed over time** (old PERMIT codes vs CPA 2.1), so long series need mapping.
- Floor area is in thousands of m2 (`UNIT_MULT=3`). Record the download date and pin `startPeriod` for reproducibility.

### Role in BuildingLens

Sector trends and context, not the core per-building feature. Concrete uses: a small trend chart (permits per canton over time), a context card next to RAG answers ("permits authorized in canton X"), and a credibility cross-check against our inspection corpus coverage. It counts as the second heterogeneous source required by the brief, clearly labeled as aggregate.

## 3. Synthetic inspection reports (Luxembourg)

No public corpus of real technical inspection reports exists at volume (they are commercial deliverables). Our strategy, stated openly:

- **Generate 20 to 50 realistic inspection-report PDFs** with a script, anchored to real Luxembourg buildings sampled from EUBUCCO LU (real footprint, height, coordinates). Reports follow the RICS C1/C2/C3 condition taxonomy and a defect vocabulary, organized by discipline.
- **Calibrate the format on 1 to 2 real public RICS sample reports** (for structure only), for example the RICS Home Survey Level 3 sample PDF.
- **Why synthetic:** there is no public per-building inspection data, and EUBUCCO has no names or addresses. Generating the corpus from code is fully reproducible (more robust than scraping) and demonstrates that we understand the target schema. In production, these would be replaced by real SECO reports without changing the pipeline.

This choice is documented honestly in the README. It is a defensible engineering decision, not a shortcut.

### Severity taxonomy

We map defect severity to a public, recognized scale:

- **RICS C1/C2/C3** condition ratings as the canonical 3-class severity (`minor` / `major` / `critical`).
- ASTM E2018 ("immediate / short-term / long-term") and Eurocode consequence classes (CC1/CC2/CC3) as additional public vocabulary cited in the README.

## Reproducibility summary

- EUBUCCO LU: anonymous S3 Parquet download, pinned to v0.2 and `nuts_id=LU00`.
- STATEC: LUSTAT SDMX API, pinned dataflows and `startPeriod`, download date recorded.
- Synthetic PDFs: generated by `scripts/generate_sample_reports.py` from a fixed seed for repeatability.

All three are public and re-creatable with `make data`. No SECO or client data is used.
