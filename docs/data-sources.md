# Data sources

BuildingLens uses public, reproducible data only. Everything here can be re-fetched from scratch by the pipeline (`make data`). We combine three real public Luxembourg sources (EUBUCCO buildings, STATEC permits, ACT commune boundaries) with a synthetic inspection-report layer, and we are explicit about what is real and what is generated.

Details below were verified on 2026-06-17.

| Source | Type | Role in BuildingLens | License | Reproducible |
|---|---|---|---|---|
| EUBUCCO v0.2 (Luxembourg) | Structured, per building | Real footprints, location, commune, footprint area; height ~78% real / ~22% ML; use and floors ML-estimated | Mixed per building: gov-luxembourg ~77% CC0, OSM ~13% ODbL, Microsoft ~10% (stored in buildings.source) | Yes, S3 download by script |
| STATEC "Autorisations de bâtir" | Structured, aggregated | Sector context: permit trends by canton and type | CC0 | Yes, LUSTAT SDMX API |
| ACT commune boundaries (LU) | Structured, polygons | Real commune per building, by point-in-polygon on the centroid | CC0 | Yes, committed GeoJSON (data.public.lu) |
| Synthetic inspection reports (LU) | Unstructured, PDF | Inspection corpus for extraction and RAG | Generated by us | Yes, generated by script |

## 1. EUBUCCO v0.2 (Luxembourg buildings)

Real building footprints for the whole country. This is our per-building anchor: it populates the `buildings` table with real geometry, height, and location.

- **What it is:** European building stock database, 200M+ individual buildings. Luxembourg is one NUTS2 region, code `LU00`. The v0.2 LU parquet we use holds 186,171 buildings (the v0.1 paper reported 143,923).
- **Publisher:** ai4up project (MCC Berlin, TU Berlin). Data paper: Milojevic-Dupont, Wagner et al., *Scientific Data* 10, 147 (2023), DOI 10.1038/s41597-023-02040-2.
- **Homepage:** https://eubucco.com
- **Docs:** https://docs.eubucco.com/v0.2/
- **License:** mixed per building, by `geometry_source`. For the LU subset: gov-luxembourg (national LOD1 cadastre) ~77% is CC0, OpenStreetMap ~13% is ODbL (attribution + share-alike), Microsoft ~10%. We store the per-building `source` so each row is attributable, and we credit EUBUCCO and OpenStreetMap. It is not a uniform CC0 dataset.

### Download (no account, anonymous S3)

Luxembourg Parquet file (geometry as WKB, EPSG:3035):

```
https://s3.eubucco.com/eubucco/v0.2/buildings/parquet/nuts_id=LU00/LU00.parquet
```

Script-friendly options: direct `curl`/`wget` on that URL, anonymous AWS CLI (`--no-sign-request --endpoint-url https://s3.eubucco.com`), GeoPandas `read_parquet` with `storage_options={"anon": True, ...}`, or a remote DuckDB `read_parquet` query. Estimated Luxembourg file size is 20 to 40 MB (derived, not officially listed).

### Fields and Luxembourg coverage

| Field | Coverage for LU | Note |
|---|---|---|
| `id` | 100% | Stable EUBUCCO id |
| `geometry` (footprint) | 100% | EPSG:3035 polygon; reproject the centroid for lat/lon, compute area in m2 from the polygon |
| `height` | 100% present | Real (cadastre / LOD1) for ~78%, ML-estimated with confidence bounds for ~22%; `height_source` distinguishes them |
| `type` / `subtype` | 100% present | ML-estimated (residential ~98,900 / non-residential ~87,200); presence is not accuracy |
| `floors` | 100% present | ML-estimated (fractional regression); we store it rounded |
| `city_id` | 100% | Commune-level code (102 values, form `LU0101`), not a city name; we resolve the commune NAME via ACT boundaries instead |
| `construction_year` | 74 / 186,171 | Essentially absent; any value is an ML estimate |
| building name | None | EUBUCCO has no names |
| address | None | EUBUCCO has no addresses |

### Mapping to the `buildings` table

| Column | From EUBUCCO | Status for LU |
|---|---|---|
| `id` | `id` | Direct |
| `latitude` / `longitude` | centroid of `geometry` reprojected to EPSG:4326 | Direct |
| `source` | `geometry_source` | Direct, per building (e.g. gov-luxembourg / osm / msft) |
| `height_m` | `height` | ~78% real (cadastre); ~22% ML-estimated (`height_source = estimated`) |
| `footprint_area_m2` | area of `geometry` (EPSG:3035) | Real, computed from the footprint |
| `use_type` / `use_subtype` | `type` / `subtype` | ML-estimated |
| `floors` | `floors` | ML-estimated, stored rounded |
| `year_built` | `construction_year` | Essentially absent (74/186,171), treat as unreliable |
| `commune` | resolved from coordinates (ACT) | Real, point-in-polygon; see source 4 |
| `name`, `address` | none | Synthetic placeholders, see caveat below |

### Key caveat: no names, no addresses, no reliable year

EUBUCCO gives us real geometry and coordinates, but **no building names, no addresses, and essentially no construction year for Luxembourg** (present on only 74 of 186,171 buildings). Height is real (cadastre) for about 78% of buildings and ML-estimated for the other ~22% (the `height_source` field flags which). Use and floors are ML-estimated. This shapes how we present building identity in the demo (documented in the README and in the synthetic-data note below). The one identity field we DO recover is the commune: we resolve it from the real centroid by point-in-polygon against the ACT commune boundaries (source 4 below), so a registry building's commune is real while its name, street, number and postcode stay synthetic.

## 2. STATEC "Autorisations de bâtir" (building permits)

Official Luxembourg construction-permit statistics. Used as **sector context**, not as per-building data.

- **Dataset page:** https://data.public.lu/en/datasets/entreprises-construction-et-logement-autorisations-de-batir/
- **Publisher:** STATEC (national statistics institute), served from the LUSTAT database.
- **License:** CC0 (public domain).

### Access: LUSTAT SDMX REST API (no key)

There are no direct CSV files on the portal; the resources point to LUSTAT. Machine-readable data comes from the SDMX REST API:

```
https://lustat.statec.lu/rest/data/LU1,{DATAFLOW_ID}/all?startPeriod={YYYY}
```

Send header `Accept: application/vnd.sdmx.data+csv;urn=true;file=true;labels=both` to get labeled CSV. Useful dataflows:

| Dataflow | Content | Level |
|---|---|---|
| `DF_D4100` | Buildings, dwellings, built volume (overview) | National, 1970-2022 |
| `DF_D4113` / `DF_D4123` | Dwellings / buildings authorized by type | Canton, 2010-2025 |
| `DF_D4131` to `DF_D4133` | Authorized useful floor area (thousands of m2) | National / regional / canton |

Granularity goes down to the 12 cantons (no commune level). Permit counts are annual; floor area is also quarterly.

### Caveats

- **Aggregated, not per-building.** Every row is a count or surface total by type, canton, and year. No permit id, address, or parcel. You cannot identify individual buildings.
- **Permits, not inspections.** Authorization to build says nothing about later inspection or defects.
- **Classification changed over time** (old PERMIT codes vs CPA 2.1), so long series need mapping.
- Floor area is in thousands of m2 (`UNIT_MULT=3`). Record the download date and pin `startPeriod` for reproducibility.

### Role in BuildingLens

Sector trends and context, not the core per-building feature. Concrete uses: a small trend chart (permits per canton over time), a context card next to RAG answers ("permits authorized in canton X"), and a credibility cross-check against our inspection corpus coverage. It counts as the second heterogeneous source required by the brief, clearly labeled as aggregate.

## 3. Synthetic inspection reports (Luxembourg)

No public corpus of real technical inspection reports exists at volume (they are commercial deliverables). Our strategy, stated openly:

- **Generate one realistic inspection-report PDF per building** with a script, anchored to real Luxembourg buildings from EUBUCCO LU (real footprint, coordinates, commune). The defects themselves are **invented from a hand-written catalogue** (`_VOCABULARY` in `src/buildinglens/synthetic_reports.py`, organized by discipline and rated on the RICS C1/C2/C3 scale), with a seeded per-building "condition" driving how many defects each report carries. So a building's location and footprint are real, but its **condition is fictional**.
- **Calibrate the format on 1 to 2 real public RICS sample reports** (for structure only), for example the RICS Home Survey Level 3 sample PDF.
- **Why synthetic:** there is no public per-building inspection data, and EUBUCCO has no names or addresses. Generating the corpus from code is fully reproducible (more robust than scraping) and demonstrates that we understand the target schema. In production, these would be replaced by real SECO reports without changing the pipeline.
- **Evaluation is a mechanics check, not real accuracy:** `eval/gold.jsonl` is emitted by the same `_report_plan` generator that writes the PDFs, so the perfect `make eval` score (241 defects across 40 buildings, P/R/F1 = 1.00) only shows the LLM re-reading what the generator injected. Expect lower numbers on real, heterogeneous reports.

This choice is documented honestly in the README. It is a defensible engineering decision, not a shortcut.

### Severity taxonomy

We map defect severity to a public, recognized scale:

- **RICS C1/C2/C3** condition ratings as the canonical 3-class severity (`minor` / `major` / `critical`).
- ASTM E2018 ("immediate / short-term / long-term") and Eurocode consequence classes (CC1/CC2/CC3) as additional public vocabulary cited in the README.

## 4. ACT commune boundaries (real commune)

Official administrative boundaries of the 99 current Luxembourg communes (the ACT file has 100 features, including a "Lac de la Haute-Sûre" lake territory we drop), used to recover the **real commune** of a building from its coordinates (point-in-polygon), since EUBUCCO carries only a coarse `city_id` code.

- **Publisher:** ACT (Administration du cadastre et de la topographie), via data.public.lu.
- **License:** CC0 (public domain).
- **Format:** GeoJSON, EPSG:4326, property `COMMUNE`. 100 features = 99 communes plus a "Lac de la Haute-Sûre" lake territory, which we drop on load so it is never assigned to a building. About 1 MB.
- **Stable URL:** https://data.public.lu/fr/datasets/r/16103fa4-7ff1-486a-88bc-5018353957ea (302s to the dated file).
- **How we use it:** committed at `data/reference/lu_communes_4326.geojson` (outside the gitignored `data/raw/`) so the build is offline and byte-for-byte reproducible. `src/buildinglens/communes.py` loads the 99 communes once into a shapely STRtree and maps each building centroid to its commune. If the file is ever missing, a deterministic synthetic grid is used and flagged via `is_synthetic`.
- **Caveat:** a centroid on a border, in a water body, or with coordinate noise can match no polygon; those buildings keep a NULL commune and are simply not returned by commune search. A random commune may still appear in the synthetic address string, but that is cosmetic and does not set the commune field. Coverage (resolved/total) is printed at build time.

## Reproducibility summary

- EUBUCCO LU: anonymous S3 Parquet download, pinned to v0.2 and `nuts_id=LU00`.
- STATEC: LUSTAT SDMX API, pinned dataflows and `startPeriod`, download date recorded.
- ACT commune boundaries: committed GeoJSON at `data/reference/lu_communes_4326.geojson` (CC0), loaded offline.
- Synthetic PDFs: generated by `src/buildinglens/synthetic_reports.py` from a fixed seed for repeatability; the same generator emits the evaluation gold set.

All four sources are public and re-creatable with `make data`. No SECO or client data is used.
